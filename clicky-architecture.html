Deep Dives — How Everything Actually Works

🔒 What is the Worker Proxy and Why Does It Exist?

The problem

Clicky needs API keys for Claude, Gemini, ElevenLabs, and AssemblyAI. If those keys were stored inside the app, anyone who downloads Clicky could extract them and run up your bill. Even if you obfuscate them, a determined person with a debugger could find them in minutes.

The solution

A Cloudflare Worker sits between the app and every external API. It's a tiny server (just ~185 lines of TypeScript) that runs on Cloudflare's edge network. The real API keys are stored as encrypted secrets on Cloudflare — the app never sees them. When the app wants to talk to Claude or Gemini, it sends the request to the Worker, which injects the correct API key and forwards the request upstream.

┌──────────┐ ┌─────────────────┐ ┌──────────────┐ │ Clicky │ ──POST──▶ │ Cloudflare Worker │ ──POST──▶ │ Claude / Gemini │ │ (no keys) │ │ (has all keys) │ │ (real API) │ └──────────┘ └─────────────────┘ └──────────────┘ Injects API key into the request headers/URL, then pipes the response straight back to Clicky

The Worker has four routes, each proxying to a different service: /chat for Claude, /gemini for Gemini, /tts for ElevenLabs text-to-speech, and /transcribe-token for getting temporary AssemblyAI tokens. Each route injects the relevant API key in whatever format that service expects (header, query parameter, etc.) and streams the response back.

🧠 How AI Vision Works

What happens when you speak

When you release the push-to-talk keys, two things happen simultaneously: your voice gets transcribed into text, and Clicky takes a screenshot of every connected monitor. These screenshots are JPEG images, each labeled with their pixel dimensions (e.g. "Screen 1 — Primary Focus (image dimensions: 2560x1600 pixels)").

The transcript and screenshots are bundled together and sent to the AI. The AI literally sees your screen the same way you'd see a photo — it can read text, identify buttons, recognize apps, see your code, spot UI elements. It then answers your question using both what you said and what it sees.

Example

You say "how do I commit this?" while looking at Xcode. The AI sees the Xcode window in the screenshot, spots the Source Control menu in the menu bar, and responds: "see that source control menu up top? click that and hit commit, or you can use command option c as a shortcut."

How the AI "sees" images

Both Claude and Gemini are multimodal models — they accept both text and images as input. The screenshots are base64-encoded (converted to a long text string representing the image bytes) and embedded directly in the API request alongside the text. The AI processes text and images together in a single pass, which is why it can reference specific things on your screen.

What the AI receives: [Image: Screenshot of Screen 1, 2560x1600px] [Image: Screenshot of Screen 2, 1920x1080px] (if multi-monitor) [Text: "how do I commit this?"] [System prompt: personality, rules, pointing instructions] [Conversation history: previous exchanges] What the AI returns: "see that source control menu up top? click that and hit commit. [POINT:285,11:source control]"
Streaming (SSE)

The AI doesn't wait until the full response is ready — it streams text back word by word using Server-Sent Events (SSE). This is the same technology that makes ChatGPT's responses appear to type themselves. Clicky receives each text chunk as it's generated and accumulates the full response. This reduces perceived latency: the TTS engine can start speaking before the full response is complete.

📍 How Cursor Pointing Works — When It Points vs. Just Speaks

The mechanism

Clicky has a small blue triangle cursor that floats on a transparent overlay above all your windows. When the AI wants to point at something on your screen, it embeds a special tag at the end of its response:

[POINT:x,y:label] — point at coordinates (x,y) with a label [POINT:1100,42:color inspector] — point at pixel (1100,42), label it "color inspector" [POINT:400,300:terminal:screen2] — point at (400,300) on the second monitor [POINT:none] — don't point at anything

The app strips this tag from the spoken text (so you don't hear "point colon 285 comma 11"), parses the coordinates, figures out which monitor they belong to, and animates the blue cursor along a bezier arc to that spot. The coordinate space matches the screenshot dimensions — so when the AI says (285, 11), it means 285 pixels from the left edge and 11 pixels from the top of the screenshot it was shown.

When does it point?

The AI decides on its own whether pointing would help. It's instructed with these guidelines in the system prompt:

Points at things when:

You ask how to do something in an app ("how do I commit?"), you're looking for a menu or button ("where's the color inspector?"), you need help navigating ("how do I open terminal?"), or the answer involves a specific UI element on your screen that would be clearer to show than describe.

Doesn't point when:

You ask a general knowledge question ("what is HTML?"), the conversation has nothing to do with your screen ("tell me a joke"), you're asking about something conceptual ("explain recursion"), or the element would be so obvious that pointing adds nothing ("the thing you're already looking at").

Examples from the actual system prompt
User asks how to color grade in Final Cut: "you'll want to open the color inspector — it's right up in the top right area of the toolbar." [POINT:1100,42:color inspector] User asks what HTML is: "html stands for hypertext markup language, it's basically the skeleton of every web page." [POINT:none] Element is on a different monitor: "that's over on your other monitor — see the terminal window?" [POINT:400,300:terminal:screen2]
Multi-monitor coordinate mapping

Each screenshot is labeled with its screen number and whether it's the "Primary Focus" (where the cursor is). The AI uses the :screenN suffix to specify which monitor the element is on. The app maps screenshot pixel coordinates to real display coordinates, accounting for the difference between Retina pixels (what the screenshot captures) and display points (where macOS positions windows). For example, on a 2x Retina display, screenshot coordinate (1000, 500) maps to display point (500, 250).

🎙 The Voice Pipeline — From Sound Waves to Text

Push-to-talk, not always-listening

Clicky only listens when you hold Ctrl+Option. This is a deliberate design choice — it never records in the background, never sends audio when you're not actively pressing the keys. The moment you release, recording stops and the audio is finalized.

The keyboard shortcut is detected by a CGEvent tap — a low-level macOS mechanism that listens for key events system-wide, even when Clicky isn't the focused app. This is more reliable than standard AppKit keyboard monitors for detecting modifier-only shortcuts.

Real-time streaming transcription

While you're speaking, the audio isn't just being recorded for later — it's being streamed live to AssemblyAI over a WebSocket connection. Your microphone audio is captured by AVAudioEngine, converted to PCM16 mono format (a simple, uncompressed audio format), and sent in real-time chunks. AssemblyAI processes these chunks as they arrive and returns partial transcripts, so by the time you release the keys, the final transcript is ready almost instantly — there's no "processing" delay.

Timeline: Ctrl+Option pressed ────────────────────────── Ctrl+Option released │ │ ▼ ▼ Mic starts ─▶ Audio chunks streamed to AssemblyAI ─▶ Final transcript (happening in real-time via WebSocket) (ready in ~100ms)

🔊 Text-to-Speech — How Clicky Talks Back

Two TTS engines

After the AI generates a response, the text (with the POINT tag stripped out) is sent to a text-to-speech engine to be spoken aloud. Clicky has two options:

ElevenLabs (primary) — A cloud TTS service that produces natural, human-sounding speech. The text is sent to the Cloudflare Worker's /tts route, which forwards it to ElevenLabs with the API key. ElevenLabs returns an MP3 audio file, which the app plays through AVAudioPlayer. The voice is configurable — Clicky uses a specific voice ID stored in the Worker's config.

macOS System TTS (fallback) — If ElevenLabs fails (invalid key, out of credits, proxy detection), Clicky falls back to NSSpeechSynthesizer, the built-in macOS speech engine. It's less natural-sounding but it's free, works offline, and requires no API key. This ensures Clicky always speaks, even when the cloud service is down.

💭 Conversation Memory

How Clicky remembers what you said

Clicky keeps a rolling history of your last 10 exchanges (your transcript + the AI's response). Every time you push-to-talk, all previous exchanges are included in the API request, so the AI has context about what you've been talking about. This is why you can say "tell me more about that" or "what about the other file?" and it knows what "that" or "the other file" refers to.

The history is capped at 10 exchanges to prevent the request from getting too large (remember, each exchange includes screenshots). It resets when you quit and reopen the app — there's no persistent storage of conversations.

What gets sent on your 4th question: Exchange 1: user: "what app is this?" ai: "that's VS Code..." Exchange 2: user: "how do I open terminal?" ai: "hit control backtick..." Exchange 3: user: "what does this error mean?" ai: "that's a null pointer..." Current: user: "how do I fix it?" + [screenshot of current screen] The AI knows "it" = the null pointer error from exchange 3

Tech Stack

SwiftUI + AppKit Frontend

SwiftUI for all views, bridged to AppKit NSPanel / NSStatusItem for the menu bar icon, floating panel, and full-screen cursor overlay. @MainActor isolation, async/await concurrency.

ScreenCaptureKit Screen Capture

macOS 14.2+ API for multi-monitor screenshots. Returns labeled JPEG images with pixel dimensions so the AI's coordinate space matches the real display.

AVAudioEngine Audio

Real-time microphone capture for push-to-talk. Converts buffers to PCM16 mono via BuddyAudioConversionSupport and streams to AssemblyAI or buffers for upload providers.

CGEvent Tap Input

Listen-only system-wide event tap for the Ctrl+Option push-to-talk shortcut. More reliable than AppKit global monitors for modifier-only key combos while backgrounded.

Cloudflare Workers Proxy

TypeScript edge worker holds all API secrets. Four routes proxy to Anthropic, Google, ElevenLabs, and AssemblyAI. Nothing sensitive ships in the app binary.

AssemblyAI Speech-to-Text

Real-time streaming transcription via WebSocket (u3-rt-pro model). Short-lived tokens fetched from the Worker proxy. Shared URLSession prevents connection pool corruption.

Claude + Gemini AI Vision

Vision models analyze screenshots + voice transcript. SSE streaming for progressive text. Claude uses content_block_delta, Gemini uses candidates[].content.parts[].

ElevenLabs / macOS TTS Text-to-Speech

Primary: ElevenLabs eleven_flash_v2_5 via Worker proxy. Fallback: NSSpeechSynthesizer — free, offline, works when ElevenLabs is unavailable.

Element Pointing Cursor

AI embeds [POINT:x,y:label:screenN] tags in responses. Parsed via regex, mapped to the correct monitor's coordinate space, animated along a bezier arc.
